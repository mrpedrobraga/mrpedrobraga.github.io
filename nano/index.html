nano is a strong-typed compiled language with a modern design that targets simplicity and strong semantical consistency.

Most importantly, nano is an art project made to be the best language design ever for the tastes of me, specifically.

If you're still interested, let's learn nano by example!

This is the hello world example. 

print "Hello World"

Ok, now here's a simple calculator CLI app. 

print 'CALCULATOR\n'

loop (
  let op = prompt 'Please input the desired operation (+, -, *, /): '

  if not op in ['+', '-', '*', '/'] then (
    print 'Operation not recognized.'; continue
  )

  with ParseErrorHandler

  let a = float::from_string(prompt('First parameter: '))
  let b = prompt('Second parameter: ') as float

  let result = match op
when '+' then a + b
when '-' then a - b
when '*' then a * b
when '/' then a / b

  print '\n{a} {op} {b} = {result}!'
)

ParseErrorHandler = err_handler (e: ParseError) -> (
  print 'Not a valid number: {e.parse_input}'
  continue
)

Pattern matching, returning error unions instead of throwing, conversions, container operators, interpolation... that's a lot of concepts, yet I bet you managed to understand all of it.

#Design Goals

The things I keep in mind while designing the language.


nano is a thin wrapper.

Nano is compiled directly into machine code. No dependencies, no hacks, no runtime. What you write is what gets compiled.

The development environment you need is simply the nano compiler nnc. When you use it, it'll create a (deletable) .nano subfolder for cache. After compilation you'll have a single executable.

There's no package manager, no boilerplate, no enforced folder structure.


nano is semantic

The biggest challenges developers face isn't speed nor fancy algorithms. It's authoring codebases that can be understood by other people, like they themselves the next day.

nano renames concepts to make them less confusing, but still trusting the inteligence of the developers to figure out how to use them. For example...

// What?
int a = 0;
int *b = &a;

a = 2
std::cout << *b  // prints 2?? WHAT?? WHY??
// THE HECK IS A '*'?
// Ok... pointers... does * create a pointer or dereference it?
# Let's see nano...
let a = slot<int>(0) # A slot of int... so a place where you put an int...
let b = a    # b refers to the same slot

a = 2    # changing the value in a... should change it in b, too!
print b  # 2!! Yeah!

This should be the expected experience while learning nano.


nano is expressive

nano started as a design for a pseudo-code language, something that you can use to quickly throw together to explain your thought process around solving some problem. Slowly I realized it could become a real language.

nano values allowing the programmer to think fluidly, and tries to offer type-safety and the like without compromising the creative flow.

It's also multi-paradigm, built on top of basic procedural programming, but with great support for functional, object-oriented, data-oriented, composition over inheritance, etc. 

let nums = [...(0..100)]
fn is_even (x) -> x % 2 == 0
let doubles_of_evens = select num * 2 for num in nums when is_even num
let last_num = doubles_of_evens::pop()
let half = fn x -> x // 2
last_num |> half _ |> _ + 1 |> print _

#Very long yet non-exhaustive list of features

You can declare symbols by using `let` and assigning to an identifier.

let a = 3

You can manually add a type to the symbol. The assigned value will be converted implicitly if possible. 

let a : float = 3
let b : string = "Hello there!"
let c : slot<bool> = yes
let d : array<uint | string> = [1, 2, "Three"]

You can create your own types via `struct` or `array.`

let User = struct {
    name: string
    age: uint
    products: list<Product>

    construct (name: string, age: uint) -> (
  # Notice I call it `self` and not `this`, like a maniac.
  self.(name, age) = (name, age)
    )
}

let victor = User("Victor", 22)

The primitive types (and your structs) are passed by value, and by copy.

let a = 3
let b = a # a is copied into b
b -= 1
a += 2
print b   # 2
print a   # 5

For your value sharing needs there are reference primitives. For example, slot.

let a = slot(4)
let b = a

b -= 1

print a  # 3

Get it? A slot is something you put things inside! The slot itself is passed around by copy, but internally it points to the same place in memory.

Reference types happen to be proxies, which mean they can be treated just like their T type in most cases... even fitting in functions with require T by automatically unwrapping their value. 

let a = slot(4)
fn f (x: int) -> print int

f(a)   # Works, 4 gets passed by copy.

A reference type or collection that's declare inside a function has two counterparts: one in the heap, one in the stack. The heap-allocated memory will be freed when the stack drops. 

let a : slot<int> = 102

With values that are passed by copy, no bigger issues arise with this but for reference types, an asynchronous function could have invalid references. 

fn caller_func -> (
  let a = slot<int>(3)

  some_async_func(a)

  # `caller_func`'s scope immediately finishes, and a is freed
  # but `some_async_func` is possibly still running with 
  # an invalid reference, that's a no-no.
)

Introducing... ownership!

Wait! Calm down! This isn't Rust! Relax!

let a = slot(3)

# If you call `a` with an async function,
# it will be given ownership of `a`, and it will
# be freed whenever the called function finishes.
some_async_func(a)

# But doesn't that mean the called function could
# have an invalid reference to `a` if `some_async_func` frees its memory?
# Yes, and that's why you can't use `a` here.
print a  # ERROR: `a` was given away and can not be used here.

In order to use the reference type again, you must make sure the caller scope outlives the called function... You can do this by awaiting it somewhere. 

let a = slot(3)

let e = some_async_func(a)

print (a) # This is okay, but the value of a is frozen as `3` and of type frozen_slot<int>
a = 3     # ERROR: `a` was borrowed and is readonly until it finishes.

await e

print (a) # You can read it fine now.
a = 4     # You can set it now :-)

Classes can inherit other classes to be natively accepted in places that require the super type, but offer polymorphism.

class Person {
  name: string
  age: int

  fn greet() -> print 'Hi, my name is {name}.'
}

class Programmer : Person {
  preferred_language: 'rust' | 'typescript'
  preferred_editor: 'vim' | 'vscode' = 'vscode'

  fn greet() -> print 'I use arch btw.'
}

fn call_greet(a: Person) -> a.greet()

call_greet( Programmer {"ThePrimagen", 400, 'typescript'} )

If, for some reason, you hate polymorphism will all your heart, my implementation of interfaces (traits) will put a smile on your face.

trait Personhood {
  name: string
  age: int

+  fn greet() -> print 'Hi, my name is {name}.'
}

class Programmer {
  has Personhood

  fn greet() -> print 'Uh oh.'
  #  ^^^^
  # ERROR: trait 'Personhood' already
  # has a non-abstract method `greet() -> string`.
}

use 'graphics.image'

# There's a specific data type, entity, which is comprised of only traits.
trait Velocity {
  shared position: fvec2
  velocity: fvec2
  
  fn process (dt: float) -> position += velocity * dt 
}

trait Render {
  # In memory, this `shared` field will be the same
  # for both Render and Velocity when they're applied
  # to the same entity.
  shared position: fvec2
  image: Image2D

  fn process () -> draw(image, position)
}

let a = entity [Velocity, Render]

a.process(1 / 60)  # Calling a method on an entity will call it on all its traits.
