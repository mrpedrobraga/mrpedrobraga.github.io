nano is a strong-typed compiled language with a modern design that targets simplicity and strong semantical consistency.

Most importantly, nano is an art project made to be the best language design ever for the tastes of me, specifically.

If you're still interested, let's learn nano by example!

This is "Hello World." 

print "Hello World"

Ok, now here's a simple calculator CLI app. 

print 'CALCULATOR\n'

loop (
  let op = prompt 'Please input the desired operation (+, -, *, /): '

  if not op in ['+', '-', '*', '/'] then (
    print 'Operation not recognized.'; continue
  )

  with ParseErrorHandler

  let a = float::from_string(prompt('First parameter: '))
  let b = prompt('Second parameter: ') as float

  let result = match op
    when '+' then a + b
    when '-' then a - b
    when '*' then a * b
    when '/' then a / b

  print '\n{a} {op} {b} = {result}!'
)

ParseErrorHandler = err_handler (e: ParseError) -> (
  print 'Not a valid number: {e.parse_input}'
  continue
)

#Design Goals

The things I keep in mind while designing the language.


nano is a thin wrapper.

nano is compiled directly into machine code. No installing, no dependencies, no hacks, no runtime.

The development environment you need is simply the nano compiler nnc. When you use it, it'll create a .nano subfolder for cache, but that can be deleted and should go in your .gitignore. After compilation you'll have a single executable that just works.

There's no package manager, no boilerplate, no enforced folder structure.


nano is semantic

The biggest challenges developers face isn't speed nor fancy algorithms. It's authoring codebases that can be understood by other people, like they themselves the next day.

nano renames concepts to make them less confusing, but still trusting the inteligence of the developers to figure out how to use them. For example...

// What?
int a = 0;
int *b = &a;

a = 2
std::cout << *b  // prints 2?? WHAT?? WHY??
// THE HECK IS A '*'?
// Ok... pointers... does * create a pointer or dereference it?
# Let's see nano...
let a = slot<int>(0) # A slot of int... so a place where you put an int...
let b = a    # b refers to the same slot

a = 2    # changing the value in a... should change it in b, too!
print b  # 2!! Yeah!

This should be the expected experience while learning nano.


nano is expressive

nano started as a design for a pseudo-code language, something that you can use to quickly throw together to explain your thought process around solving some problem. Slowly I realized it could become a real language.

nano values allowing the programmer to think fluidly, and tries to offer type-safety and the like without compromising the creative flow.

That's why it is multi-paradigm, built on top of basic procedural programming, but with great support for functional, object-oriented, data-oriented, composition over inheritance, etc. 

# Always expressive
let nums = [...(0..100)]
fn is_even (x) -> x % 2 == 0

# Declarative
let doubles_of_evens = select num * 2 for num in nums when is_even num

# Imperative
let last_num = doubles_of_evens::pop()

# Nested
print (last_num // 2 + 1)

# Sequential
last_num |> _ // 2 |> _ + 1 |> print _

nano is simple, but...

...not too simple. It values internal elegance, reusing keywords whenever possible, and having features that can be used in many ways but it doesn't shy away from adding features that make the coding experience better.

You probably won't learn all of nano in a weekend, and that's okay, because you don't need to to start coding some good nano.

nano is humble.

As an art-project made by a 20yo in their first job as a backend developer, nano doesn't aim to take over the world, or claim it's the best language ever. Instead, it acknowledges the worth of the classic languages (C, BASIC, FORTRAN, Java)... but with the benefit of hindsight, it's able to make a few better choices.

Syntax

nano's syntax is designed by taking into account two quirky facts:

  1. High-level programmers really like clean, simple code.
  2. Low-level programmers are much more tolerant to clutter.
nano is a multi-level language, that allocates its grammar in a way to make doing simple tasks easy, while allowing the developer to take over the wheel when and where necessary.

Symbols

You can declare symbols by using `let` and assigning to an identifier.

let a = 3

You can manually add a type to the symbol. The assigned value will be converted implicitly if possible. 

# int `3` becomes a float.
let a : float = 3

# the "Hello there!" strings just gets assigned.
let b : string = "Hello there!"

# the boolean `yes` gets converted to `slot<bool>(true).`
let c : slot<bool> = yes

# The `int`s inside the array get converted to `uint`.
let d : array<uint | string> = [1, 2, "Three"]

Types

You can create your own types via `struct` or `array`.

let User = struct {
    name: string
    age: uint
    products: list<Product>

    construct (name: string, age: uint) -> (
      # Notice I call it `self` and not `this`, like a maniac.
      self.(name, age) = (name, age)
    )
}

let victor = User("Victor", 22)

The primitive types (and your structs) are passed by value, and by copy.

let a = 3
let b = a # a is copied into b
b -= 1
a += 2
print b   # 2
print a   # 5

Reference Types

For your value sharing needs there are reference primitives. For example, slot.

let a = slot(4)
let b = a

b -= 1

print a  # 3

Get it? A slot is something you put things inside! The slot itself is passed around by copy, but internally it points to the same place in memory.

Reference types happen to be proxies, which mean they can be treated just like their T type in most cases... even fitting in functions with require T by automatically unwrapping their value. 

let a = slot(4)
let f = fn (x: int) -> print int

f(a)   # Works, 4 gets passed by copy.

Functions

Oh, look! You've just seen a function! A function in nano is something that encapsulates reusable 'behaviour,' or a reusable expression.

let reverse_of = fn (a: string): string -> (
  a.as_reverse()
  # Functions are first-class citizens.
  # If you look back to a few code blocks ago,
  # you'll see struct (and class) definitions are first-class too.
)

# But hey, look at this cool shorthand for definition
# and implementation of a function.
fn anti_reverse_of (a: string): string -> reverse_of(reverse_of(string))

# Functions are very cute-looking :)
# If no type is specified it will be inferred from usage.
let square = fn x -> x * 2

###
Then you can use it with any type that implements '*'.

(NERD ALERT!)
More specifically, the type of x is inferred to be an anonymous trait
that implements a $"*" function with an integer on the right.

fn (x: T is WithOperator<$"*", int>) -> (
  x * 2
)

The return type of the function is inferred from usage, too.

ReturnTypeOf<T::$"*".when(right: int)>
###

fn fa () -> (
  let a = 3
  let fb = fn -> print a  # Closures store the value by copy
  a = 4

  return fb
)

let c = fa()()  # '3'

# You may invoke a function passing named parameters.
fn some_big_fn (a: string, b: string, c: f32 | bool, d: option<bool>) -> ...
some_big_fn(
  a: "Hey"
  b: "There" # Named parameters!
  c: false
)

# Since parenthesis are not required on function calls,
# you might use a single struct parameter as well.
fn some_other_big_fn (options: FunctionOptions) -> ...
some_other_big_fn {
  a: "Hey"
  b: "There"
  c: false
}

fn fc (...args: array<int>) -> (print args)

fn(3, 5, 5) # variable argument count

let some_list = [1, 2, 3, 4]

fn (0, ...some_list, 5) # spread container into arguments

You can create a function that acts on a central entity (what I call a 'method') like such:

fn double (self: int) -> self = self * 2

let b = 40
b::double()
# ^^^^ an external method is called with `::` to differentiate it from a member.
# This makes compilation faster since `nnc` doesn't have to check your entire codebases
# for methods every time it finds a member access `.`.

let a = 30
double(self: a)   # <- It's the equivalent of this function call with a named `self` parameter. 

# Methods on builtins can't alter them.
# Create a method for a reference type instead.

fn double_in_place (self: slot<int>) -> self *= 2

let b = slot(30)
b::double_in_place()
print b # '60'

That's because asynchronous functions might return a state type that allows `awaiting` for its completion. It is called `Expectation.` 

let e: Expectation<int> = get_some_int()

let i: int = await e  # You can directly await an expectation.

A reference type that's declared inside a function has two counterparts: one in the heap, one in the stack. The heap-allocated memory will be freed when its associated stack counterpart is freed on a stack frame drop. 

fn main() -> (
  let a : slot<int> = 102
  a += 2
  print a

  # Scope ends, stack gets dropped, `a` gets freed.
)

This way of freeing memory is no issue with values passed by copy, nor reference types in normal functions.

But the combination of reference types and asynchronous functions is complicated!

fn caller_func -> (
  let a = slot<int>(3)

  some_async_func(a)

  # `caller_func`'s scope immediately finishes, and `a` is freed
  # but `some_async_func` is possibly still running with 
  # an invalid reference. That's a no-no.
)

Reference Ownership

Wait! Calm down! This isn't Rust! Relax!

let a = slot(3)

# If you call `a` with an async function,
# it will be given ownership of `a`.
# `a` now gets freed whenever `some_async_func` finishes. 
some_async_func(a)

# But doesn't that mean the caller function could
# have an invalid reference to `a` if `some_async_func` finishes early?
# Yes, and that's why you can't use `a` in this scope anymore.
print a  # ERROR: `a` was given away and can not be used here.

In order to use the symbol again, you must make sure the caller scope outlives the called function... You can do this by `awaiting` the called function somewhere. 

let a = slot(3)

let e = some_async_func(a)

print (a) # This is okay, but the value of a is frozen as `3` and of type frozen_slot<int>
a = 3     # ERROR: `a` was borrowed by `some_async_func` and is readonly until it returns.

await e

print (a) # You can read it fine now.
a = 4     # You can set it now :-)

The concept of nothing.

There's no such thing as null or undefined.

# A function may return an error union
let a = float::from_string("0.5") # returns float!ParseError

# or an option<T>
let myoption = get_some_float()   
if myoption.present then (
  print myoption.value
)

# You can only use the value of an option
# when it is present (it narrows the type of `option<T>::value` to `T`.)

struct option<T> {
  present: bool
  value: if present is true then T else discard

Error Handling

You may add to a scope, an error handler, which watches out for errors err'd by functions.

err_handler ParseErrorHandler <T>(e: ParseError) -> (
  print 'The input {e.parse_input} is not a valid number.'

  # This return here returns the original scope
  # the error handler is in...
  return
)

with ParseErrorHandler

# Because the error handler exits when it finds an error,
# the value of what's be a float!ParseError union gets narrowed!!!
let a : float = float::from_string("banana")

use 'graphics.image'

# There's a specific data type, entity, which is comprised of only traits.
trait Velocity {
  shared position: fvec2
  velocity: fvec2
  
  fn process (dt: float) -> position += velocity * dt 
}

trait Render {
  # In memory, this `shared` field will be the same
  # for both Render and Velocity when they're applied
  # to the same entity.
  shared position: fvec2
  image: Image2D

  fn process () -> draw(image, position)
}

let a = entity [Velocity, Render]

a.process(1 / 60)  # Calling a method on an entity will call it on all its traits.

Annotations are `incredibly` powerful.

# `%%compile_time`, as a builtin annotation, supports labels.

# Define an annotation for a function that runs only when
# `ncc compile` is called with a 'test' label
#%alias %%test as %%compile_time(labels: ['test'])

fn add (x, y) -> x + y

%%test fn -> (
  let result = add(3, 5)
  print 'TEST: add(3, 5) -> {result}'

fn mul (x, y) -> x + y
fn div (x, y) -> x / y

%%test fn arithmetic -> (
  %%assert_eq add2(3, 5), 8
  %%assert_eq sub(3, 5), -2
  %%assert_eq mul(1, 2), 2
  %%assert_eq div(3, 2), 1.5
  %%assert_errs div(4, 0)
)

But that's not all, annotations are useful when writing your 'actual code,' too.

They can manipulate the code you write and add extra functionality while avoiding clutter.

# Annotation that makes a function log its
# return value when called in runtime.

%%annotation fn log (target_ast: slot<NanoFunctionDeclarationNode>) -> (
  let current_fn: NanoFunctionDeclarationNode = target_ast;

  target_ast = %%ast fn <T>(...args: T) -> (
    (%%splat_ast current_fn)(...args) |> print _ |> return _
  )
)

# Usage:
%%log
fn add (a: int, b: int) -> a + b

add(2, 3)   # prints '5'
